BonsaiPress — Systemkritik und Empfehlungen für das Rewrite
============================================================
Datum: 2026-06-13


1. ZUSAMMENFASSUNG
------------------
BonsaiPress ist ein funktionsfaehiges Multi-Client-CMS mit einem klaren Konzept:
gemeinsamer CMS-Kern in cms/, isolierte Client-Projekte als Git-Repos in clients/,
statischer HTML-Export als Deployment-Ziel. Das Grundkonzept ist solide. Die
Implementierung hat jedoch strukturelle Probleme, die ein TDD-basiertes Rewrite
zwingend adressieren muss.


2. KONKRETE BUGS UND SCHWACHSTELLEN IM AKTUELLEN CODE
------------------------------------------------------

2.1 bp create ist unvollstaendig und GitHub-abhaengig
    cms/bin/src/Project.php::create() fuehrt keine git-Operationen aus.
    Es echot nur Befehle auf stdout und schliesst mit sleep(1).
    Der Befehl hat keinen Effekt.
    Zusaetzlich ist die Template-URL hardcoded:
        'git clone https://github.com/Agundur-KDE/emptyContent.git current'
    Das System kann kein neues Projekt anlegen ohne GitHub-Zugriff.
    Eigene Git-Server (Gitea, GitLab, self-hosted) werden nicht unterstuetzt.

2.2 bp clone hat einen Tippfehler
    cms/bin/src/Bonsai.php::clone() sucht nach dem Verzeichnis 'repro/'
    statt 'current/'. Der Befehl wuerde auf jedem System fehlschlagen.

2.3 Unsicheres git checkout in bp switch
    Project.php::switch() ruft direkt passthru("git checkout $version") auf.
    Die $version kommt aus einer Datei im Client-Repo (bootstrap-version.txt).
    Kein Escaping, kein Whitelisting. Beliebige Shell-Injektion moeglich.

2.4 Deploy-Mechanismus (hashme.php): Konzept korrekt, Implementierung fehlerhaft
    Das Prinzip ist beabsichtigt und sinnvoll: lokale Hashes werden mit den
    Server-Hashes verglichen, um auch serverseitige Aenderungen zu erkennen
    (direkter Edit ohne git push, Manipulation durch Dritte usw.).
    Das Problem liegt nicht im Konzept, sondern im fehlenden Cleanup:
    Wenn _create_hashes() fehlschlaegt, wird _deleteHashme() nie aufgerufen
    und hashme.php bleibt dauerhaft auf dem Produktivserver liegen.
    Fix: try/finally sicherstellen dass _deleteHashme() immer ausgefuehrt wird.

2.5 $_SESSION im CLI-Kontext
    Publish::makeStatic() greift auf $_SESSION['lang'] zu. Im CLI-Kontext
    (bp static) ist keine Session vorhanden. Das fuehrt zu einem Notice und
    undefiniertem Verhalten.

2.6 Globale Konstanten als Abhaengigkeiten
    Wesentliche Pfade (_path_to_content_, _path_to_includes_) sind PHP-Konstanten,
    die beim Bootstrap gesetzt werden. Jede Klasse die sie nutzt, ist implizit
    an den globalen Zustand gebunden und nicht isoliert testbar.

2.7 Passwort im Klartext im Git-Repo
    current/config/ecms_config.php (Client-Repo) enthaelt FTP-Zugangsdaten
    im Klartext, z.B. ECMS_CONFIG::$ftp_passwd = '...'.
    Diese Datei ist ins Client-Git-Repo eingecheckt. Jeder mit Repo-Zugriff
    sieht die Produktiv-Credentials. Beim Rewrite: Secrets in .env-Datei
    auslagern, .env in .gitignore aufnehmen.


3. STRUKTURELLE PROBLEME (TESTBARKEIT)
---------------------------------------

3.1 ECMS_CONFIG: statische Properties als globaler State
    Das Config-System besteht aus zwei Schichten:
    - cms/include/ecms_config.php: Defaultwerte als public static $x = '...'
    - current/config/ecms_config.php: Pro-Projekt-Overrides per ECMS_CONFIG::$x = '...'
    Das Pattern ist konzeptuell in Ordnung. Das Problem fuer TDD:
    Static Properties sind globaler mutierbarer State, der zwischen Tests leckt.

    Test A setzt ECMS_CONFIG::$domain = 'test-a.de'.
    Test B laeuft danach im selben Prozess und bekommt 'test-a.de' statt des Defaults.

    Das eigentliche Problem ist tiefer: alle Klassen rufen ECMS_CONFIG::$x direkt
    auf (Publish, PageObject, class_ftp usw.). Die Config ist hart verdrahtet,
    nicht injiziert. Ein Test kann nicht sagen "teste Publish mit dieser Config".

    Loesung fuer das Rewrite:
        interface Config {
            public function domain(): string;
            public function ftpHost(): string;
            // ...
        }
        class EcmsConfig implements Config { /* liest ECMS_CONFIG::$* */ }
        class TestConfig implements Config { /* nimmt array im Konstruktor */ }

    Das bestehende Zwei-Schichten-Pattern (Default + Override) kann intern in
    EcmsConfig erhalten bleiben. Nach aussen sieht jede Klasse nur das Interface.

3.2 Keine Klasse ist testbar
    Alle zentralen Klassen (PageObject, Publish, Project, Bonsai) sind entweder
    voll von statischen Methoden oder haengen von globalem State ab:
    - $_SESSION, $lang als globale Variable
    - registry::get() / registry::set() als globaler Container
    - Konstanten statt injizierten Pfaden
    Ein Unit-Test kann keine dieser Klassen isoliert instanziieren.

3.3 Shell und Filesystem sind nicht abstrahiert
    shell_exec(), passthru(), exec(), file_put_contents(), copy() werden direkt
    in Business-Logik aufgerufen. Fuer Tests muesste das Dateisystem und die
    Shell real vorhanden sein. Integration-Tests sind dadurch sehr aufwendig,
    Unit-Tests unmoeglich.

3.4 FTP-Logik sitzt mitten in der Deploy-Klasse
    class_ftp.php wird direkt in Publish instanziiert. Kein Interface, keine
    Abstraktion. Ein Test von Publish::_ftp_sync() wuerde eine echte FTP-
    Verbindung benoetigen.

3.5 Gemischte Verantwortlichkeiten
    Publish::makeStatic() macht gleichzeitig: Session lesen, XML parsen,
    PageObject aufrufen, Filesystem schreiben, Sitemap generieren, Link-
    Rewriting. Jede Verantwortung ist eine eigene Testdomaene.


4. EMPFEHLUNGEN FUER DAS REWRITE
----------------------------------

4.1 Dependency Injection konsequent einsetzen
    Jede Klasse erhaelt ihre Abhaengigkeiten im Konstruktor. Keine Globals,
    keine Konstanten in Business-Logik, kein Singleton-Registry.

    Beispiel:
        class SiteRenderer {
            public function __construct(
                private PageRepository $pages,
                private TemplateEngine $tpl,
                private ProjectContext $ctx,
            ) {}
        }

4.2 Shell, Filesystem und FileTransfer hinter Interfaces
    Fuer Tests: Fake-Implementierungen. Fuer Produktion: echte Implementierungen.

        interface Shell        { public function run(string $cmd): Result; }
        interface Filesystem   { public function write(string $path, string $content): void; }
        interface FileTransfer { public function put(string $local, string $remote): void; }

4.3 XML-Seitenstruktur ersetzen
    SimpleXML mit XPath ist umstaendlich zu nutzen und zu testen.
    Empfehlung: SQLite (file-based, kein Server, gut querybar) hinter einem
    PageRepository-Interface. Die Baumstruktur bleibt erhalten.

        interface PageRepository {
            public function findById(int $id): Page;
            public function findAll(): iterable;
            public function getRoot(): Page;
        }

4.4 Content-Type-System formalisieren
    Das Plugin-Konzept ist gut. Mit einem Interface wird es testbar und
    erweiterbar ohne Konventionen auswendig lernen zu muessen.

        interface ContentType {
            public function id(): string;
            public function render(Page $page, RenderContext $ctx): string;
        }

4.5 CLI mit Symfony Console
    Jeder bp-Befehl wird eine eigene Command-Klasse mit definierten Inputs
    und Outputs. Symfony Console ist klein, PSR-konform und macht Commands
    einzeln testbar.

4.6 Static Generation sauber entkoppeln
    Ein StaticExporter ohne Session-State:
        1. Alle Seiten aus PageRepository laden
        2. SiteRenderer::render(Page) aufrufen
        3. Ergebnis ueber Filesystem-Interface schreiben
        4. Sitemap separat generieren

4.7 Deploy-Mechanismus: Konzept behalten, Implementierung absichern
    Der hashme.php-Ansatz ist korrekt (bidirektionale Pruefung: lokal vs. Server).
    Er erkennt auch serverseitige Aenderungen die nie in git gelandet sind.
    Konkrete Verbesserung: try/finally garantiert dass hashme.php immer geloescht
    wird, auch bei Fehlern in Zwischenschritten.
    Optional: bei gleichzeitigen Deploys einen Lock-Mechanismus erwaegen
    (z.B. Lock-Datei auf dem Server), da web_hashes.json sonst ueberschrieben
    werden kann.


4.8 bp create vollstaendig implementieren, Template konfigurierbar machen
    Der Workflow fuer ein neues Projekt ist:
        1. Empty-Content-Template klonen (Basis-Struktur)
        2. .git entfernen
        3. Neues (privates) Git-Repo initialisieren und verknuepfen
        4. clients/<NAME>/.repository.txt anlegen

    Die Template-URL darf nicht hardcoded sein. Sie soll in einer
    BonsaiPress-weiten Konfiguration stehen (z.B. ecms/config.php oder
    eine eigene bonsai.ini) und auf jeden beliebigen Git-Server zeigen:
    GitHub, GitLab, Gitea, self-hosted SSH-Repo usw.

    Konfigurationsbeispiel:
        [bonsai]
        template_repo = git@git.example.com:templates/empty-content.git

    bp create <NAME> fuehrt alle Schritte automatisch aus:
        1. git clone <template_repo> current/
        2. rm -rf current/.git
        3. git init current/
        4. Neue Repo-URL interaktiv abfragen (einzige Benutzereingabe)
        5. git remote add origin <neue-repo-url>
        6. clients/<NAME>/ anlegen
        7. echo <neue-repo-url> > clients/<NAME>/.repository.txt
        8. Erster Commit + git push

    Kein manuelles Nacharbeiten. Nach bp create ist das Projekt
    sofort mit bp switch <NAME> aktivierbar.

4.9 Config hinter Interface kapseln
    Das bestehende Pattern (Default-Klasse + Pro-Projekt-Override) ist gut und
    kann intern erhalten bleiben. Fuer TDD muss die Config injizierbar sein:

        interface Config {
            public function domain(): string;
            public function ftpHost(): string;
            public function publishMethod(): string;
            // ...
        }

        class EcmsConfig implements Config {
            // kapselt intern ECMS_CONFIG::$* + Override-Mechanismus
        }

        class TestConfig implements Config {
            public function __construct(private array $values) {}
            public function domain(): string { return $this->values['domain']; }
        }

    Secrets (FTP-Passwort, htaccess-Credentials) kommen aus einer .env-Datei
    die in .gitignore steht und nie ins Repo eingecheckt wird.


5. TDD-STRATEGIE
-----------------

Layer                    | Test-Typ    | Hilfsmittel
-------------------------|-------------|----------------------------------
PageRepository           | Unit        | SQLite in-memory
SiteRenderer             | Unit        | FakeFilesystem, FakePage
ContentType-Plugins      | Unit        | FakeRenderContext
bp switch / bp static    | Integration | temporaeres Verzeichnis, echtes git
HTTP-Routing             | Feature     | eingebetteter PHP-Server (PHPUnit)
Deploy (FTP-Sync)        | Integration | FakeFileTransfer

Reihenfolge:
    1. PageRepository + Page-Modell (Kern des Systems)
    2. SiteRenderer + ContentTypes
    3. StaticExporter
    4. CLI-Commands (bp switch, bp static, bp deploy)
    5. Deploy/FTP-Sync


6. I18N — MEHRSPRACHIGKEIT IM REWRITE
--------------------------------------

6.1 Aktueller Ansatz und seine Grenzen
    Im aktuellen System wird pro Sprache ein eigenes Projekt angelegt
    (z.B. Agundur-DE und Agundur-EN als separate Git-Repos).
    Grund: die Seitenstruktur kann sich zwischen Sprachen unterscheiden —
    andere Navigation, andere Seiten, andere Hierarchie.
    Nachteil: Config, Templates und Assets muessen doppelt gepflegt werden.
    Deployments muessen fuer jedes Sprachprojekt separat ausgefuehrt werden.

6.2 Zielmodell: Sprachen innerhalb eines Projekts (Modell A)
    Ein Projekt = ein Kunde. Sprachen sind Varianten innerhalb des Projekts.

    Verzeichnisstruktur:
        clients/Agundur/
          config/            <- gemeinsam: FTP, Domain, ecms_config
          templates/         <- gemeinsam
          assets/            <- gemeinsam (Bilder, CSS, JS)
          languages/
            de/              <- eigene Seitenstruktur + Content
            en/              <- eigene Seitenstruktur + Content

    Jede Sprache hat ihren eigenen vollstaendigen Seitenbaum.
    Keine erzwungene 1:1-Entsprechung zwischen den Sprachen.
    Eine Seite die nur auf Deutsch existiert ist ausdruecklich erlaubt.

6.3 Optionale Verknuepfung zwischen Sprachvarianten
    Seiten koennen eine alternate_lang-Referenz auf die entsprechende Seite
    in einer anderen Sprache deklarieren:

        page id=42 lang=de  -->  alternate: id=17 lang=en

    Wird genutzt fuer:
    - hreflang-Tags im HTML-Head (SEO)
    - Sprachumschalter im Template ("diese Seite auf Englisch")
    Fehlt eine Verknuepfung, zeigt der Umschalter auf die Startseite
    der anderen Sprache.

6.4 Static Generation mit Mehrsprachigkeit
    bp static generiert HTML fuer alle Sprachen in einem Durchlauf.
    Ausgabe-Struktur (konfigurierbar per Projekt):

    Variante 1 — Sprachpfad:
        /de/index.html
        /en/index.html

    Variante 2 — Sprache als Subdomain (in ecms_config):
        agundur.de/index.html        (de, Hauptdomain)
        en.agundur.de/index.html     (en, Subdomain)

6.5 PageRepository mit Sprachkontext
    Der Sprachkontext wird beim Aufbau des Repositories uebergeben,
    nicht als globale Session-Variable:

        $repo = new SqlitePageRepository($dbPath, language: 'de');
        $repo = new SqlitePageRepository($dbPath, language: 'en');

    Fuer Tests: TestPageRepository mit hartkodiertem Inhalt, unabhaengig
    von Sprache oder Dateisystem.


7. FAZIT
--------
Der groesste Gewinn beim Rewrite liegt nicht in einer neuen Technologie, sondern
in konsequentem Dependency Injection und Interfaces um Shell, Filesystem und
FileTransfer. Erst dann wird TDD praktisch moeglich. Das bestehende Konzept
(Multi-Client, statischer Export, Git-basierter Workflow) ist gut und sollte
erhalten bleiben.
